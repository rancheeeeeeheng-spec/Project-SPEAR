# Database Design

## Project S.P.E.A.R.

---

# Overview

Microsoft Dataverse serves as the central data platform for Project S.P.E.A.R., providing a secure, relational database that supports business processes, workflow automation, and reporting.

Rather than storing information across multiple disconnected data sources, Dataverse provides a **single source of truth** for operational data, ensuring consistency, integrity, and scalability.

The database design emphasizes normalization, modularity, and maintainability while supporting future expansion.

---

# Design Objectives

The database was designed with the following objectives:

* Centralize operational data
* Eliminate duplicate records
* Support workflow automation
* Maintain data integrity
* Enable secure access control
* Simplify reporting
* Support future scalability
* Provide a maintainable schema

---

# Design Principles

Project S.P.E.A.R. follows several database design principles:

* Single source of truth
* Relational data modelling
* Minimal data redundancy
* Normalized structure
* Clear entity relationships
* Secure data access
* Extensible schema
* Enterprise scalability

# Core Tables

## Personnel

Stores information relating to application users.

### Typical Attributes

* Personnel Identifier
* Display Name
* Email Address
* Status
* Assigned Appointment
* Role
* Last Updated

Purpose:

Acts as the primary reference for all personnel-related activities throughout the application.

---

## Appointment

Represents operational appointments and assignments.

### Typical Attributes

* Appointment Identifier
* Appointment Name
* Assignment Status
* Assigned Personnel
* Effective Date

Purpose:

Provides a structured method for assigning personnel to operational responsibilities.

---

## Recall

Stores recall activities initiated within the application.

### Typical Attributes

* Recall Identifier
* Recall Status
* Start Time
* End Time
* Created By
* Recall Type

Purpose:

Acts as the central record for operational readiness events.

---

## Attendance

Stores attendance information linked to personnel and operational activities.

### Typical Attributes

* Attendance Identifier
* Personnel
* Event
* Attendance Status
* Timestamp

Purpose:

Provides historical attendance records for reporting and operational review.

---

## Check-In

Stores QR code attendance transactions.

### Typical Attributes

* Check-In Identifier
* Personnel
* Check-In Time
* Validation Status

Purpose:

Records attendance events initiated through the application.

# Relationships

The solution uses relational links between business entities.

| Parent      | Relationship | Child             |
| ----------- | ------------ | ----------------- |
| Personnel   | 1 : Many     | Attendance        |
| Personnel   | 1 : Many     | Recall Responses  |
| Personnel   | 1 : Many     | Check-In          |
| Appointment | 1 : Many     | Personnel History |
| Recall      | 1 : Many     | Attendance        |
| Recall      | 1 : Many     | Notifications     |

This relational structure reduces duplication while simplifying reporting and automation.

---

# Primary Keys

Each table contains a unique identifier generated by Microsoft Dataverse.

Examples include:

* Personnel ID
* Appointment ID
* Recall ID
* Attendance ID
* Check-In ID

These identifiers are managed by Dataverse and ensure uniqueness across the solution.

---

# Lookup Relationships

Lookup columns establish relationships between business entities.

Examples:

* Personnel → Appointment
* Attendance → Personnel
* Attendance → Recall
* Check-In → Personnel
* Recall → Created By

Using lookups improves data consistency and simplifies application logic.

---

# Choice Columns

Choice columns standardize frequently used values.

Examples include:


### Attendance Status

* Present
* Pending
* Absent

### Recall Status


* Active
* Completed


Using predefined choices improves reporting consistency and reduces data entry errors.

---

# Data Integrity

Several mechanisms help maintain data quality.

Examples include:

* Required fields
* Lookup validation
* Choice columns
* Business rules
* Duplicate prevention
* Relationship enforcement

These controls reduce inconsistent or incomplete records.

---

# Security Model

Access to data is controlled using Microsoft Dataverse security capabilities.

The design supports:

* Role-based access
* Table-level permissions
* Record ownership
* User authentication via Microsoft Entra ID
* Least privilege principles

Sensitive information is protected through platform-level security rather than custom application logic.

---

# Reporting Considerations

The database structure has been designed to support reporting without requiring extensive data transformation.

Typical reporting scenarios include:

* Personnel statistics
* Attendance summaries
* Recall participation
* Operational readiness metrics
* Administrative workload
* Historical trends

The relational design enables efficient querying and dashboard generation.

---

# Scalability

The schema supports future expansion through additional entities and relationships.

Potential future additions include:

* Training Records
* Equipment Assignment
* Incident Reporting
* Task Management
* Operational Exercises
* Asset Tracking

These can be incorporated without restructuring the existing data model.

# Best Practices Applied

The database design follows Microsoft Dataverse best practices by:

* Using relational tables instead of duplicated data
* Leveraging lookup relationships
* Applying standardized choice columns
* Separating business entities by responsibility
* Supporting solution-aware deployment
* Designing for extensibility
* Maintaining a scalable schema

The Project S.P.E.A.R. database has been designed as a secure, scalable, and maintainable enterprise data model.
